## Reprise rule

At every new reprise, before opening work:

1. Read `README.md`.
2. Read `START_HERE.md`.
3. Read `CHANGELOG.md`.
4. Read `release_notes/v0.1.1.md`.
5. Check open PRs.
6. Check whether any issue, reader feedback, factual error, or boundary risk exists.
7. Decide whether the next move is:
   - NO_ACTION
   - MINOR_DOC_CLARIFICATION
   - RELEASE_PREP
   - PDF_PREP
   - EXTERNAL_ANNOUNCEMENT_PREP
   - COMMERCIALIZATION_PREP
   - STOP_RISK

Default decision is NO_ACTION unless a concrete trigger exists.

## Valid triggers for a new front

A new PR is justified only if at least one of these is true:

- real reader confusion has been observed
- GitHub issue or external feedback points to a problem
- factual error is found
- boundary wording is unsafe or overclaiming
- release metadata is inconsistent
- attribution or license wording is wrong
- a deliberate PDF pass is explicitly opened
- a deliberate external announcement pass is explicitly opened
- a deliberate commercial packaging pass is explicitly opened

No new PR should be opened merely because the repo can be polished further.

## Rule for future reprise traces

Every meaningful repository movement must leave enough trace for a future operator to answer:

- What changed?
- Why did it change?
- What did not change?
- What boundaries remain?
- What is the next allowed state?
- What is still not opened?

That trace does not always require a dedicated checkpoint file.

Acceptable reprise traces include:

- clear PR body
- changelog entry
- release note
- short closeout note
- repo checkpoint
- issue or comment decision record

Create a dedicated checkpoint only when the movement changes one of these:

- public status
- release posture
- publication posture
- PDF posture
- commercial posture
- boundary posture
- reader-facing authority
- evidence / review / approval distinction
- permission-to-act wording

Do not create checkpoints merely because a minor wording change was made.

Checkpoint density must remain minimal.

Strict method does not mean bureaucratic overhead.

The right standard is:

- sufficient proof
- minimal trace
- clear boundary
- no overclaim
- no accidental capability opening
- no governance replacing progress
