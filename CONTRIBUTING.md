# Contributing

AI Ops SOP Pack is a public Markdown documentation pack.

Contributions should keep it practical, conservative, and evidence-first.

## Principles

1. **Reader first**
   - A first-time reader should understand what the pack is and what it is not.
   - Prefer plain language before internal language.
   - Link SOPs, templates, and examples clearly.

2. **Evidence before readiness**
   - Do not upgrade a status because work feels close.
   - Readiness follows observed evidence.
   - Review, approval, and permission-to-act must stay separate.

3. **Documentation, not authority**
   - This pack is not a runtime.
   - This pack is not a permission system.
   - This pack does not grant merge, provider, trading, publishing, or client-contact authority.

4. **Adaptation required**
   - Teams must replace commands, branches, repository names, CI names, paths, and approval gates with their own environment.
   - Examples are illustrative only.

5. **Keep it small**
   - Prefer concise SOPs, templates, examples, and diagrams.
   - Do not turn the pack into a framework or tool.

## Good contributions

Good contributions include:

- clearer public explanations;
- safer boundary language;
- better templates;
- realistic examples that remain explicitly fictitious or environment-scoped;
- stronger distinction between evidence, review, approval, and action;
- diagrams or tables that improve understanding.

## Avoid

Avoid:

- production-readiness claims;
- certification claims;
- sales claims;
- live runtime claims;
- hidden implementation assumptions;
- environment-specific commands presented as generic;
- dependencies unless clearly justified;
- examples that imply authority without evidence.

## Final check

Before adding a change, ask:

```text
Does this make the pack easier to understand, adapt, or review without creating false authority?
```

If not, do not add it.
